Back to the Data Model. 
  • Statement of User Requirements
A Database is needed to support a Debt Collection business.

The focus is entirely on the debtors.
Our Clients have the Derbtors and we provide services to our Clients to manage the Debtors.

As soon an process is open we start collect all kind of information about the Debtors :-

  • Confirmation of their location

  • if they have more business

  • have more debts etc.

This focus on the Debtors reflects the way the business processes are organized.

A client which contracts our services receives a contract number -
  - like 2170001 - 21 is telephone code of Lisbon, 7 from 2007 and 0001 is the number of the contract.



  • After the contract been approved by our Office... what happens?


  • Each Debt has its own Contract.

but one Contract can be associated with 10 debts or 50 debts.

So each debt of the same contract is identified by the 21-7-0001-0901 keeping the same 0001 for the 10 or 50 debts,
only the last four or five digits grows - identify the debt from others debts we have:
21700010901
21700010902
21700010903
21700010904
21700010905 and so on

So if the tables has information regarding the Debtors how to get info to give to client ?

We have this number (21700010900) to assure an simple query identifies all the processes regarding the same client:
217-0001 the middle number is always the same for the same client.
If the client is other then the middle's change to 0002...

So an Query can get all the processes registred with the same contract number,
and produced in a report to send quickly to the client...

This is how we organize the processes and i hope this explanation simplifies what we want and the DB Design.

We suggest also that it is important to pay close attention to how the Factoring Institutions organize their Database.
Ours will probably be quite similar except we don't do credit but our clients which factoring does.
