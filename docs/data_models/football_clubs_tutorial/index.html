If you are new to Data Models, this page of my new Tutorial will help you understand the basics of a Data Model. 

There are three simple rules we will follow :-
1. Identify the things we are Interested in
2. Decide how they are related
3. Establish what else we know about them.
  
Rule number 1. Identify the things we are Interested in
1) We decide these are
Clubs, Players and Games
 
1) The things we are Interested are
Clubs, Players and Games.
 
2a. Decide how they are related
Clubs employ Players and play Games against other Clubs.
 
2b. Decide how they are related (continued)
So far so good - except that when we think about how this works in practice, we realise
that not all Players play in every Game.
Therefore we need a more flexible way of identifying which players play in each Game.

In  addition , a Player will usually play in the same position.
But changes are possible and to provide for these chamges, we need to identifdy the position each Player
plays in during each Game.
When we think about Positiins, we realise that they sare a small number of predefined Positions and therefore,
we should prefix the 'Positions' Entity name by 'Ref_' to establish that this is Reference Data.


So our Business Rules become :-
Clubs employ Players who play in Games against other Clubs.
The specific Players and their Position in each Game will be decided before each Game.
 

3. Establish what else we know about them
The additional data items that we kow about the Entities is shown in this 3NF Data Model which shows all Attributes
 
Dimensional Model
This shows a Dimensional Model which is used to provide data for Business Intelligence (BI) 
Data Marts with Conformed Dimensions
This is an alternative to the Dimensional Model shown above.
In this approach we separate the data in the Dimensional Modelinto Games and Playeers and each has its own Data Mart.
THen we identify the Dimensions that they share and show them between the two Data Marts.
We cal these 'Conformed Dimensions' because the values have to conform to the same values in each Data Mart.
If they do not, then the values cannot be used to join Data Sets and the results obtained would be complete nonsense.
Putting it differently, if the shared dimensions are not Conformed then our work as Data Warehouse designers and
BI specialists would be completely wasted and we should take up a different line of work.
The best and simplest example of a Conformed Dimension is dates, which we show in a Calendar.
In passing, I call my Calendar entity 'Ref_Calendar' to emphasise the fact that it is Reference Data.

Dates in a calendar are an example of Master Data, and by definition, Master Data will always be Conformed Dimensions
because they represent one consistent value across an organisation.

In our case, w can use the Games Data Mart to obtain statistics about the Club and its standing in the League.

The Players Data Mart, of course, would be used to obtain individual statistics about our favourite players and to
identify, for example, who is the most prolific scorer.
